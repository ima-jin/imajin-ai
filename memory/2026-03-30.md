## What was built

Implemented two-part DFOS relay bridging so Imajin identities (did:imajin:*) get their genesis operations ingested into the gossip relay (relay.relay_identity_chains), enabling federation with Brandon's ATX/NYC/LIS peered relays.

### Part 1 — New registrations (going forward)

The existing registration flow already accepted an optional `dfosChain` payload from the client (browser-side signed using `@imajin/dfos`'s `createIdentityChain`). It called `storeDfosChain()` which wrote to `auth.identity_chains` only.

**Change:** `storeDfosChain()` now also calls `ingestToRelay(chain.log)` immediately after writing to the local DB. The relay receives the JWS tokens and enters the gossip network.

### Part 2 — Lazy backfill on login

**Change:** After successful key-challenge verification in `POST /api/login/verify`, an async (non-blocking, non-fatal) relay backfill fires:

1. Look up local chain from `auth.identity_chains` via `getChainByImajinDid()`
2. If found, check relay with `checkRelayChain(dfosDid)` → `GET REGISTRY_URL/relay/identities/{did}`
3. If not on relay, call `ingestToRelay(chain.log)` to submit

This backfills all 128 existing identities that already bridged their chain locally but were never relayed.

## Key files changed

- `apps/auth/lib/dfos.ts` — added `ingestToRelay()`, `checkRelayChain()`; modified `storeDfosChain()` to relay after local store
- `apps/auth/app/api/login/verify/route.ts` — added async relay backfill block after successful auth
- `apps/auth/.env.local` — added `REGISTRY_URL=http://localhost:3002`
- `apps/auth/.env.example` — added `REGISTRY_URL=http://localhost:3002`

## Decisions made

- **No server-side genesis signing**: The server never holds private keys. Genesis operations must come from the client (via `body.dfosChain`). Server only verifies + relays.
- **Relay URL**: `REGISTRY_URL/relay/operations` (registry is on port 3002; relay is mounted at `/relay/*`)
- **Identity lookup endpoint**: `GET /relay/identities/{dfosDid}` returns 404 if not found, 200 if present
- **All relay calls are non-fatal**: Relay being down must never break registration or login

## Current status

Implementation complete. Requires `REGISTRY_URL` to be set in auth's environment (local: `http://localhost:3002`). The relay bridge is forward-compatible — any user who has bridged (dfosChain in auth.identity_chains) will be automatically backfilled on their next login.

## Part 3 — DFOS Genesis Creation (issue #532 part 2)

### Problem

128 existing identities have no DFOS chain at all (nothing in `auth.identity_chains`, nothing on the relay). New registrations also don't create a DFOS genesis unless the client sends one. We can't create genesis operations server-side — the server never holds private keys.

### Solution: client-initiated genesis + server-side hint

The server already accepted `body.dfosChain` in both register and login/verify. The missing piece was **telling clients when they need to create a chain**.

**Changes made:**

1. **`apps/auth/lib/dfos.ts`** — added `hasDfosChain(imajinDid)` helper: lightweight check that returns `boolean` (queries only the `did` column from `identity_chains`).

2. **`apps/auth/app/api/login/challenge/route.ts`** — challenge response now includes `hasDfosChain: boolean`. When `false`, the client knows to: (a) call `createIdentityChain({privateKeyHex, publicKeyHex})` from `@imajin/dfos`, then (b) include the result as `dfosChain` in the `/login/verify` body.

3. **`apps/auth/app/api/login/verify/route.ts`** — response now includes `dfosChainLinked: boolean` (true = newly stored this request; false = none submitted or already existed).

4. **`apps/auth/app/api/register/route.ts`** — all register response paths now include `dfosChainLinked: boolean`.

### Backfill flow for existing 128 identities

On next login, the challenge response returns `hasDfosChain: false`. A client that reads this flag creates the genesis chain locally (private key required) and includes it in the verify call. The server verifies, stores, and ingests to relay. No server-side key handling needed.

### Key design decision

No server-signed genesis option was implemented — the `@metalabel/dfos-protocol` `signIdentityOperation` requires a private key signer, which the server cannot provide. Option (a) — client creates genesis — is the correct architecture. The `packages/dfos/src/bridge.ts` `createIdentityChain()` function is the canonical implementation; `apps/auth/lib/dfos-client.ts` wraps it for browser use.

## Part 4 — Password login DFOS genesis (issue #532 part 3)

### Problem

`PasswordAuthTab.tsx` was the only login flow NOT creating DFOS chains. The register page and `KeyAuthTab.tsx` both already called `createDfosChain()`. Password login was the gap.

### What was built

**`apps/auth/app/login/components/PasswordAuthTab.tsx`** — modified `handlePasswordSubmit()`:

1. Now destructures `hasDfosChain` from the challenge response (alongside `challengeId` and `challenge`)
2. If `hasDfosChain` is false, dynamically imports `createDfosChain` from `@/lib/dfos-client` and calls it with `{ privateKey: privateKeyHex, publicKey: publicKeyHex }` — both are available in scope at that point (private key decrypted from stored key, public key derived via `ed.getPublicKeyAsync`)
3. Chain creation is wrapped in try/catch — non-fatal, login proceeds regardless
4. `dfosChain` (or `null`) is always included in the `/api/login/verify` body — server ignores null

### Other login flows checked

- **`KeyAuthTab.tsx`** — already creates DFOS chain (lines 91-94), passes in `/api/register` body ✓
- **`MfaGate.tsx`** — no DFOS handling needed; chain is submitted in the preceding `/api/login/verify` call before MFA gate activates ✓
- **Register page** — already creates chain (lines 141-148) ✓

### Current status

All client-side login/register paths now create and submit DFOS genesis chains when `hasDfosChain` is false. The full backfill path for existing 128 unchained identities is now complete end-to-end.
