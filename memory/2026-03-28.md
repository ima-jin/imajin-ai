## feat(auth): Auth Method Management + MFA UI (#493)

Branch: `feat/493-auth-mfa-ui`

### What was built

**Backend — new/modified API routes:**
- `GET /api/account/methods` — public endpoint; takes `?handle=` or `?did=`, returns `{ did, hasStoredKey, encryptedKey?, salt?, keyDerivation?, mfaMethods }`. Rate-limited 20 req/min.
- `POST /api/login/verify` — modified: after sig verification, checks active MFA methods (verifiedAt IS NOT NULL); if any, returns `{ mfaRequired: true, methods, challengeToken }` (5-min EdDSA JWT) instead of creating session.
- `POST /api/login/mfa` — new: verifies `challengeToken`, validates TOTP or email code, creates session cookie.
- `POST /api/mfa/email/send` — new: sends 6-digit code via `@imajin/email`; stores in `lib/email-mfa-codes.ts` in-memory Map.
- `POST /api/mfa/totp/disable` — new: requires auth + valid TOTP code; deletes mfaMethods record.

**Backend — new libs:**
- `lib/email-mfa-codes.ts` — in-memory TTL Map for email MFA codes.
- `lib/jwt.ts` — added `createMfaChallengeToken` / `verifyMfaChallengeToken` (5-min, type: 'mfa_challenge').

**Frontend — login page refactor:**
- `app/login/page.tsx` — stripped to tab container + MFA gate overlay; removed email magic link flow.
- `app/login/components/KeyAuthTab.tsx` — challenge-response flow (`/api/login/challenge` + `/api/login/verify`), falls back to `/api/register` for new identities; MFA-aware.
- `app/login/components/PasswordAuthTab.tsx` — two-step: handle lookup → PBKDF2+AES-GCM client decrypt → challenge-response; MFA-aware.
- `app/login/components/MfaGate.tsx` — 6-digit input with auto-submit on 6th digit; method selector; email send button; trust-device checkbox.

**Frontend — settings:**
- `app/settings/security/page.tsx` — full security hub: auth methods status, TOTP setup/disable with QR flow, email/SMS toggles (email functional, SMS stub), session duration picker, known devices list with trust/remove.

### Architecture note
Client-side crypto uses Web Crypto PBKDF2-SHA256 (310,000 iterations) + AES-256-GCM. `keyDerivation: 'pbkdf2'`. No new browser deps.

Challenge signing: `TextEncoder().encode(challengeHex)` — matches server's `verifySignature` in `lib/crypto.ts`.

Email MFA codes are stored in-memory (process-local); fine for single-process; stateless after server restart.

## feat: Scaffold notify service + caller package (#479)

Branch: `feat/479-notify-service`

### What was built

**`apps/notify/`** — New Next.js service on port 3008 (dev) / 7008 (prod), schema: `notify`

**Database schema (`src/db/schema.ts`):**
- `notifications` table — stores per-user notifications with recipient DID, scope, urgency, title, body, jsonb data, channels_sent array, read/readAt
- `preferences` table — per-user per-scope email/inapp toggle settings

**API routes:**
- `POST /api/send` — internal webhook (x-webhook-secret auth); resolves template, checks prefs, stores notification, sends email via `@imajin/email`
- `GET /api/notifications` — paginated list (newest first, cursor via `?before=<id>`)
- `GET /api/notifications/unread` — unread count
- `POST /api/notifications/:id/read` — mark one read
- `POST /api/notifications/read-all` — mark all read
- `GET /api/preferences` — list all prefs for DID
- `PUT /api/preferences/:scope` — upsert preference (email/inapp toggles)
- `GET /api/health` — health check

**Templates (`src/templates/index.ts`):** 7 scope templates: `market:sale`, `market:purchase`, `event:ticket`, `event:registration`, `coffee:tip`, `coffee:tip-sent`, `connection:invite-accepted` — each with title/body functions and email HTML using `emailWrapper` from `@imajin/email`

**Settings page (`app/settings/page.tsx`):** Dark-theme preferences UI with toggle switches grouped by service (Market, Events, Coffee, Connections)

**`packages/notify/`** — Thin caller package (`@imajin/notify`)
- `send()` function — fire-and-forget POST to `/api/send` with webhook secret; silently skips if env vars not set
- Same pattern as `emitAttestation` in `@imajin/auth`

### Architecture note
Email sending requires `data.email` in the notification payload (recipient email address). Caller services must pass this when they want email delivery. Phase 2 could resolve this from the profile service instead.

## feat(ui): Notification bell + provider + toast integration (#479 phase 3)

Branch: `feat/479-notify-ui`

### What was built

**`packages/ui/src/notification-provider.tsx`** — `NotificationProvider` + `useNotifications` hook
- Polls `NEXT_PUBLIC_NOTIFY_URL/api/notifications/unread` every 30s
- Skips toasting on first load (sets baseline); diffs on subsequent polls
- On count increase: fetches `/api/notifications?limit=1`, fires `toast.info` (normal) or `toast.warning` (urgent) for genuinely new id
- Warns once to console if service is down; shows 0 unread otherwise
- `markAsRead(id)` → POST `/:id/read`; `markAllAsRead()` → POST `/read-all`; `refresh()` fetches last 20
- `useNotifications()` returns empty fallback (no throw) when used outside provider

**`packages/ui/src/notification-bell.tsx`** — Bell icon + dropdown panel
- Inline SVG bell icon; red badge capped at "99+"
- Panel opens on click, loads notification list via `refresh()`; closes on click-outside or Escape
- Each item: unread dot (space always reserved), title, body preview, relative timestamp (`relativeTime()`)
- "Mark all as read" button; empty state copy; `bg-gray-900 border border-gray-700` panel styling

**`packages/ui/src/nav-bar.tsx`** — Bell wired into NavBar
- Imported `NotificationBell`; rendered in `hidden sm:flex` area between quick-access icons and auth section
- Gated on `process.env.NEXT_PUBLIC_NOTIFY_URL` — bell never renders if env var absent

**`packages/ui/src/index.ts`** — New exports: `NotificationProvider`, `useNotifications`, `NotificationBell`, `Notification` type, `NotificationContextValue` type

### Architecture note
`NotificationProvider` calls `useToast()` internally (via a toastRef to avoid dep churn on the polling interval). Must be rendered inside `ToastProvider`. If `NEXT_PUBLIC_NOTIFY_URL` is unset, provider renders children with zero state and no polling — fully safe to include unconditionally at app root.
